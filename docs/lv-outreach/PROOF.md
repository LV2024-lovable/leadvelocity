## 2. Goedgekeurde scenario-cases

**Regel:** scenario's zijn **fictief-maar-plausibel** — elk moet expliciet worden gelabeld als *"voorbeeld"* of *"scenario"*. Nooit doen alsof dit echte klantcases zijn.

### 2.1 Groothandel scenario

```yaml
scenario_id: scenario_groothandel_30m
label: "Voorbeeld: technische groothandel €30M omzet"
context:
  - 80 FTE technische groothandel in industriële onderdelen
  - €30M omzet, 25.000 SKU's
  - brutomarge 27%, netto marge 5,5%
before_state:
  - "Pricing in Excel, sinds jaren niet herzien"
  - "Inkoop op gut feel, substantieel dead stock"
  - "Binnendienst tikt RFQ's handmatig, 45 min per stuk"
  - "Offerte-conversie rond 24%"
after_state:
  - "AI-pricing live op core-SKU's"
  - "Forecasting halveert nee-verkoop op runners"
  - "Quote-AI verwerkt 80% van RFQ's binnen 2 min"
  - "Pipeline-scoring wijst ondergeleverde accounts aan"
impact_statement: "Structureel hogere brutomarge, binnendienst wint uren per dag terug, offerte-conversie stijgt"
disclaimer: "Illustratief scenario op basis van sector-benchmarks. Werkelijke resultaten verschillen per bedrijf."
use_in:
  - outreach touch 4 (value-add)
  - /groothandel landing page
  - whitepaper groothandel
```

## 3. Regels voor het citeren van benchmarks/cases

```yaml
rules:
  - "Benchmark altijd gekoppeld aan bron — ook in verkorte vorm ('McKinsey, 2024')"
  - "Scenario altijd expliciet gelabeld 'voorbeeld' of 'scenario'"
  - "Nooit claimen als eigen klantresultaat totdat we 3 echte cases hebben"
  - "Nooit meerdere benchmarks in één alinea stapelen (wordt overspam)"
  - "Max 1 benchmark + max 1 scenario per outreach-mail"

when_to_use:
  - outreach_touch_1_opener: "Alleen zeer kort en alleen als opener-hook (bijv. '7% heeft AI-beleid — valideert dat jullie voorlopen')"
  - outreach_touch_1_body: "1 relevante benchmark voor credibility"
  - outreach_touch_4_value: "1 scenario uitdiepen"
  - linkedin_post: "Benchmarks prima, cases labelen als voorbeeld"
  - website: "ResultsNew sectie bevat de hoofd-benchmarks. Vertical-pages gebruik scenarios"

when_NOT_to_use:
  - "Als het signaal zelf al sterk genoeg is (dan voegt benchmark niets toe, wordt afleiding)"
  - "Bij touch 5 (break-up) — toon moet menselijk/afsluitend zijn, niet data-overloaded"
```

## 4. Hoe nieuwe benchmarks/cases toevoegen

```yaml
proces:
  - Nieuwe benchmark gevonden → check: primary source? 2024-2026? independent? → voeg toe aan juiste sectie
  - Nieuwe case (eigen klant) → pas toevoegen als schriftelijke goedkeuring van klant binnen is → verplaats van scenario naar "echte case"
  - Verlopen benchmark (>3 jaar) → mark_as_dated en vervangen door nieuwere
  - Gezond houden: review heel bestand elk halfjaar, verwijder wat niet meer citabel is
```
